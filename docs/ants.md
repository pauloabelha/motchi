# Ant Architecture

Motchi separates body drives from action policy.

## BaseAnt

`BaseAnt` is abstract. It owns the organism runtime:

- MuJoCo/Gymnasium environment lifecycle
- energy state
- energy pickup items
- recharge zone, placed away from the ant spawn by default
- perception
- active sensing cost
- drive computation
- pickup consumption
- recharge handling
- reset handling
- viewer markers
- viewer HUD overlay
- telemetry

Subclasses must implement:

```python
def choose_action(self, drives: DriveSnapshot) -> ActionCommand:
    ...
```

The `drives` snapshot is always computed before action selection. A subclass may use it, partially use it, or ignore it.

## Action Commands

Policies propose movement. They do not own body energy economics. A policy returns an `ActionCommand`:

```python
ActionCommand.from_motor(motor, label="random")
```

An action command contains:

- proposed motor output
- a label for telemetry/debugging

Energy cost is not a policy property. `BaseAnt` owns a `MotorActuator`, and the actuator computes the cost of executing a command from the motor effort. As energy gets low, motor output is smoothly scaled down. At zero energy, no motor output reaches the body.

`MotorActuator` is built from low-level motor units. Each unit has an `energy_multiplier`, so a different leg or joint can be more efficient without changing the policy:

```json
{
  "name": "front_left_hip",
  "energy_multiplier": 0.5
}
```

The same action command costs less when it passes through a more efficient actuator unit.

The boundary is:

```text
Policy        -> proposes ActionCommand
MotorActuator -> computes per-unit cost and scales execution
Energy drive  -> spends or replenishes body energy after execution
```

## RandomAnt

`RandomAnt` inherits from `BaseAnt`.

It ignores its drives and samples random motor actions:

```python
class RandomAnt(BaseAnt):
    def choose_action(self, drives: DriveSnapshot) -> ActionCommand:
        del drives
        return ActionCommand.from_motor(self.action_space.sample(), label="random")
```

Even though the action policy is random, the internal body loop still updates:

- energy depletion
- recharge detection
- pickup sensing
- pickup consumption
- dominant drive telemetry

That makes RandomAnt a useful baseline body: it has energy needs, but no intentional behavior.

## MuJoCo HUD

`BaseAnt` writes a small status overlay into the MuJoCo viewer. The HUD is per ant and currently appears in the top-right corner.

It shows:

- ant name
- energy value and bar
- dominant drive
- recharge drive and distance
- pickup drive, distance, and remaining pickup count
- sensing scale and sensing energy spent
- action scale and energy spent on the last executed command

The HUD is generated by `BaseAnt._hud_lines(...)` and rendered through the Gymnasium MuJoCo viewer's `add_overlay(...)` API.

## Reset Behavior

Ant-v5 normally terminates when the torso becomes unhealthy, such as falling below or above its healthy height range. That causes an immediate reset, which looks like the ant suddenly teleports back to the start pose.

Motchi disables this by default:

```json
{
  "environment": {
    "terminate_when_unhealthy": false
  }
}
```

This keeps the body simulation running even if random actions make the ant fall. Resets still happen for Motchi-level conditions such as energy depletion, and reset logs include an explicit reason.

To restore Gymnasium's built-in unhealthy reset behavior:

```bash
python test_ant.py --terminate-when-unhealthy
```

## Active Sensing

Sensing is not free. `BaseAnt` spends a small amount of energy before each live drive computation. The cost is configured separately from actuator energy:

```json
{
  "sensing": {
    "recharge_range": 12.0,
    "food_range": 10.0,
    "detection_threshold": 0.08,
    "base_cost": 0.005,
    "object_cost": 0.001
  }
}
```

The current model is intentionally simple and inspectable:

```text
raw_strength = 1 - (distance / sense_range)^2
signal_strength = raw_strength * sensing_energy_scale
detected = signal_strength >= detection_threshold
```

This means nearby objects are sensed strongly, distant objects fade quadratically, and weak signals disappear below threshold. If energy is too low to fully pay the sensing cost, `sensing_energy_scale` drops and perception degrades.

The recharge zone can still physically recharge the ant when the ant is inside it, even if sensing is too weak to detect it. Detection is perception; contact/recharge is world interaction.

## Config

The first ant config is:

```text
configs/random_ant.json
```

It declares:

```json
{
  "ant_type": "RandomAnt",
  "inherits": "BaseAnt"
}
```

The Python config types live in:

```text
motchi/body/config.py
```

The config is split into:

- `ViewerConfig`
- `RunConfig`
- `EnergyConfig`
- `FoodConfig`
- `ActuatorConfig`
- `SensingConfig`
- `CoreDriveConfig`
- `AntConfig`

## Adding A New Ant

Create a new subclass:

```python
from motchi.body.base_ant import BaseAnt
from motchi.runtime.actions import ActionCommand
from motchi.runtime.core_drives import DriveSnapshot


class MyAnt(BaseAnt):
    def choose_action(self, drives: DriveSnapshot):
        # Use drives.recharge_drive, drives.food_drive, or ignore them.
        return ActionCommand.from_motor(self.action_space.sample(), label="my_policy")
```

This keeps future behavior systems, memory systems, learning policies, and procedural controllers separate from the body’s internal drives.
